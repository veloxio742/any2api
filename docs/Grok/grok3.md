  **`__cf_bm` deep investigation (COMPLETED):**
  - User wanted to understand how `__cf_bm` is generated
  - We traced the full lifecycle through browser DevTools:
    1. `__cf_bm` is generated by Cloudflare Turnstile challenge (not simple JS computation)
    2. When `__cf_bm` expires/missing → request gets 403 → triggers Turnstile challenge
    3. Turnstile loads from `challenges.cloudflare.com/cdn-cgi/challenge-platform/h/g/turnstile/...`
    4. It attempts a "Private Access Token challenge" (GET request that returned 401)
    5. On success, Cloudflare server issues new `__cf_bm` via `Set-Cookie` response header

## 核心问题：Grok 没有公开 API

Grok（grok.com）只有网页版，没有像 OpenAI 那样的 `api.openai.com/v1/chat/completions` 公开 API。但很多工具（Cursor、ChatBox、LobeChat、Claude Code）都只认 OpenAI 格式的 API。

所以 grok2api 做的事情就是：**假装自己是 OpenAI API 服务器，收到请求后偷偷去 grok.com 网页版发消息，把回复翻译成 OpenAI 格式返回。**

这就是"Web To API"逆向的本质。

---

## 为什么用 curl_cffi 而不是 requests？

这是整个项目最关键的技术选型。

当你用 Python `requests` 或 Java `HttpClient` 访问 grok.com 时，Cloudflare 会返回 403。原因是 **TLS 指纹**。

HTTPS 连接建立时，客户端发送一个 ClientHello 消息，里面包含：
- 支持的加密套件列表
- TLS 扩展列表
- 扩展的顺序

不同的 HTTP 客户端（Chrome、Firefox、Python requests、Java HttpClient）发出的 ClientHello 长得完全不一样。Cloudflare 会检查这个指纹——如果不像浏览器，直接 403。

`curl_cffi` 的 `impersonate="chrome136"` 参数会让它的 TLS 握手指纹和 Chrome 136 完全一致，Cloudflare 就认为这是真浏览器。

这就是为什么代码里到处都是：
```python
session.post(..., impersonate=browser)
```

---

## 为什么只用 conversations/new？

Grok 有两个聊天端点：
- `POST /rest/app-chat/conversations/new` — 创建新对话
- `POST /rest/app-chat/conversations/{id}/responses` — 在已有对话里继续

我们之前在浏览器里测试过，`responses` 端点需要 `__cf_bm` cookie（Cloudflare Turnstile 生成的，30 分钟过期，无法程序化获取），而 `conversations/new` 不需要。

所以 grok2api 选择**每次都用 conversations/new**，代价是没有真正的多轮对话——它把历史消息全部拼接成一段文本塞进 `message` 字段。这是一个工程上的妥协：放弃原生多轮对话能力，换取不依赖 `__cf_bm`。

---

## 为什么要把 messages 数组拼成一段文本？

OpenAI 格式是这样的：
```json
{
  "messages": [
    {"role": "system", "content": "你是助手"},
    {"role": "user", "content": "你好"},
    {"role": "assistant", "content": "你好！"},
    {"role": "user", "content": "天气怎么样"}
  ]
}
```

但 Grok 的 `conversations/new` 只接受一个 `message` 字符串。所以 `MessageExtractor` 做的事情是：

```
system: 你是助手

user: 你好

assistant: 你好！

天气怎么样
```

注意最后一条 user 消息**不加角色前缀**——因为 Grok 会把整个 message 当作用户的输入，最后一条就是"当前问题"，加了 `user:` 前缀反而会让模型困惑。

这个设计看起来粗暴，但实际效果不错，因为大语言模型本身就擅长理解这种带角色标记的文本格式。

---

## 为什么要伪造 Statsig ID？

```python
message = "e:TypeError: Cannot read properties of undefined (reading 'childNodes')"
return base64.b64encode(message.encode()).decode()
```

Grok 前端集成了 Statsig SDK（一个 A/B 测试和分析平台）。每次请求都会带一个 `x-statsig-id` header，值是 base64 编码的客户端标识。

Grok 前端的 Statsig SDK 在初始化时会捕获 JS 错误并生成这个 ID。如果你不带这个 header，请求可能被拒绝。

grok2api 的做法很聪明：随机生成一个看起来像真实 JS 错误的字符串，base64 编码后当作 Statsig ID。服务端不会验证这个错误是否真的发生过，只要格式对就行。

动态生成（每次不同）比静态值更安全，因为如果所有请求都用同一个 ID，容易被识别为自动化流量。

---

## 为什么要双 Token 池？

Grok 有两种账号：
- Basic（普通 X 账号）：每个周期 80 次请求
- Super（X Premium+ 订阅）：每个周期 140 次，且能用 `grok-4-heavy` 等高算力模型

单个账号配额很快就用完了。所以 grok2api 维护两个 Token 池，每个池里放多个账号的 SSO Cookie。

选择策略是**配额最多优先**——这样可以均匀消耗所有账号，避免某个账号被集中使用导致快速耗尽。

429（配额用完）时自动换下一个 Token 重试，401（Token 失效）连续 5 次标记为 expired。这就是为什么有 `ACTIVE → COOLING → EXPIRED` 状态机。

---

## 为什么图片生成用 WebSocket？

这是 grok2api 里一个很有意思的发现。

Grok 网页版的图片生成不是通过普通 HTTP 请求返回的，而是通过 WebSocket 连接 `wss://grok.com/ws/imagine/listen` 实时推送图片数据。

流程是：
1. 建立 WebSocket 连接
2. 发送一个 JSON 消息（包含 prompt、比例等参数）
3. 服务器推送多个图片帧：preview（小预览）→ medium（中等质量）→ final（最终高清图）
4. 通过图片字节大小判断是 preview/medium/final

为什么不用 HTTP？因为 Grok 的图片生成是渐进式的——先给你一个模糊预览，然后逐步清晰化。WebSocket 天然适合这种实时推送场景。

代码里还有一个 `_BlockedError` 机制：如果收到了 medium 质量的图但迟迟没有 final 图，说明可能被 NSFW 审查拦截了。这时会启动并行补偿——同时发多个请求，只要有一个成功就返回。

---

## 为什么要 FlareSolverr？

`cf_clearance` 是 Cloudflare 挑战通过后颁发的 Cookie。有些情况下（IP 被标记、地区敏感），即使 TLS 指纹对了，Cloudflare 还是会要求过挑战。

FlareSolverr 是一个 Docker 容器，里面跑着真正的 Chrome 浏览器。它会：
1. 用 Chrome 访问 grok.com
2. 自动通过 Cloudflare Turnstile 挑战
3. 把获得的 `cf_clearance` Cookie 和 User-Agent 返回给你

grok2api 定时调用 FlareSolverr 刷新 `cf_clearance`，这样即使 Cookie 过期也能自动恢复。

关键点：**获取 cf_clearance 时用的 User-Agent 必须和后续请求的 User-Agent 完全一致**，否则 Cloudflare 会拒绝。这就是为什么 FlareSolverr 返回的 UA 会被保存并用于所有后续请求。

---

## 为什么响应解析那么复杂？

Grok 返回的不是标准 SSE（`data: {json}`），而是逐行 JSON：

```json
{"result":{"response":{"token":"你"}}}
{"result":{"response":{"token":"好"}}}
{"result":{"response":{"isThinking":true,"token":"让我想想..."}}}
{"result":{"response":{"modelResponse":{"outputText":"你好！"}}}}
```

`StreamProcessor` 需要处理：
1. 普通 token → 直接转成 OpenAI 的 `delta.content`
2. `isThinking: true` → 包裹在 `<think>` 标签里（思维链）
3. `streamingImageGenerationResponse` → 图片生成进度
4. `modelResponse` → 最终完整响应（包含图片 URL）
5. `cardAttachment` → 卡片附件
6. `<xai:tool_name>` / `<xai:tool_args>` → 工具调用（Grok 原生不支持 function calling，通过 prompt 注入 + XML 解析模拟）

每种情况都要正确转成 OpenAI SSE 格式。这就是为什么 `StreamProcessor` 有那么多状态变量（`think_opened`、`image_think_active`、`_tool_state` 等）。

## 总结一下设计哲学

整个项目的核心思路就是：

1. **伪装成浏览器** — curl_cffi 模拟 TLS 指纹，Headers 模拟 Chrome 请求
2. **避开最难的反爬** — 只用 `conversations/new` 避开 `__cf_bm`，用 FlareSolverr 自动过 Cloudflare 挑战
3. **格式翻译** — OpenAI messages 数组 ↔ Grok 单条文本，OpenAI SSE ↔ Grok 逐行 JSON
4. **高可用** — 多 Token 池 + 自动切换 + 配额同步 + 定时刷新

每个设计决策都是在"能用"和"成本"之间找平衡。比如放弃原生多轮对话换取不依赖 `__cf_bm`，用 FlareSolverr 的 Docker 开销换取自动过 Cloudflare，用 prompt 注入换取 tool calling 支持。

`cf_clearance` 之所以能跨环境用，是因为它的校验相对宽松，主要看 UA 和 IP。但 `__cf_bm` 是更细粒度的会话级 Cookie，校验可能更严格。

**所以 grok2api 的选择是合理的：** 与其花大量资源去维护一个可能不稳定的 `__cf_bm` 刷新机制，不如直接绕开它 — 只用 `conversations/new`，用文本拼接模拟多轮对话。稳定性远比"真多轮"重要。

## 思路 1：共享浏览器 Cookie 池（最实用）

不用 FlareSolverr 那种"每次启动新浏览器过挑战"的方式，而是保持一个**长期运行的浏览器实例**，专门负责维护 Cookie。

具体做法：用 Playwright 启动一个 Chrome，登录 Grok 账号，然后这个浏览器就一直开着。写一个定时任务，每 20 分钟让浏览器随便访问一下 grok.com（比如加载首页），Cloudflare 会自动刷新 `__cf_bm`。然后从浏览器的 Cookie Store 里把最新的 `__cf_bm` + `cf_clearance` 提取出来，写到一个共享存储（Redis / 文件 / 内存）。

API 服务发请求时，从共享存储里拿最新的 Cookie 用。

```
┌─────────────────┐     Cookie Store      ┌─────────────────┐
│  Playwright      │ ──── 写入 ────────→  │  Redis / File    │
│  (长期运行)       │   __cf_bm            │                  │
│  每20分钟刷新     │   cf_clearance       │                  │
└─────────────────┘                        └────────┬────────┘
                                                     │ 读取
                                            ┌────────▼────────┐
                                            │  API 服务        │
                                            │  curl_cffi 发请求 │
                                            └─────────────────┘
```

优点：`__cf_bm` 是在同一个浏览器环境里自然刷新的，指纹一致性没问题。一个浏览器实例可以服务所有 API 请求。

缺点：需要维护一个浏览器进程，但比每个请求都开浏览器轻量得多。

**关键问题：`__cf_bm` 跨环境能用吗？** 这个需要实测。如果 Cloudflare 只校验 Cookie 值 + UA + IP，那从 Playwright 拿到的 `__cf_bm` 用在 curl_cffi 里应该没问题（保持 UA 一致）。如果 Cloudflare 还校验 TLS 指纹和 Cookie 的绑定关系，那就不行了。

## 思路 2：Playwright 代理模式（最可靠）

不用 curl_cffi 发请求了，所有请求都通过 Playwright 浏览器发出。但不是每个请求都开一个浏览器，而是用 **CDP（Chrome DevTools Protocol）** 直接控制浏览器发 fetch 请求。

```java

    const response = await fetch('/rest/app-chat/conversations/{id}/responses', {
        method: 'POST',
        headers: {...},
        body: JSON.stringify(payload)
    });

""");
```

这样所有请求都是从真实浏览器环境发出的，TLS 指纹、Cookie、`__cf_bm` 全部自然维护，Cloudflare 完全看不出是自动化。

而且这样就能用 `conversations/{id}/responses` 端点了，真正的多轮对话。

缺点：
- 并发受限于浏览器标签页数量（一个 Chrome 实例大概能开 20-50 个标签页）
- 内存占用大（一个 Chrome 实例 200-500MB）
- 需要解析浏览器返回的响应，流式处理比较麻烦

但如果你的并发量不大（比如个人使用或小团队），这个方案其实很合适。

---

## 思路 3：逆向 Turnstile 挑战（最硬核）

直接搞清楚 `__cf_bm` 的生成算法，用纯代码生成。

我们之前在浏览器里追踪过，`__cf_bm` 是 Cloudflare Turnstile 脚本生成的。Turnstile 脚本从 `challenges.cloudflare.com` 加载，是混淆过的 JS。理论上可以：

1. 下载 Turnstile 脚本
2. 反混淆
3. 找到生成 `__cf_bm` 的逻辑
4. 用 Node.js / Java 复现

但这条路极难：
- Turnstile 脚本经常更新，混淆方式会变
- 它会收集浏览器环境信息（Canvas、WebGL、字体等），你需要模拟整个浏览器环境
- Cloudflare 专门有团队对抗这种逆向
- 即使你今天搞定了，下周可能就失效了

不推荐，除非你对 Cloudflare 反爬有深入研究。

---

## 思路 4：混合架构（最平衡）

结合思路 1 和 grok2api 现有方案：

- **短对话**（1-3 轮）：用 `conversations/new` + 文本拼接，不需要 `__cf_bm`，和现在一样
- **长对话**（4+ 轮）：切换到 Playwright 代理模式，用 `conversations/{id}/responses`，真多轮

这样大部分请求走轻量的 curl_cffi 路径，只有真正需要长上下文的对话才走浏览器路径。

```
请求进来
  │
  ├─ messages ≤ 3 轮
  │   → curl_cffi + conversations/new（快，轻量）
  │
  └─ messages > 3 轮
      → Playwright + conversations/{id}/responses（慢，但真多轮）
```

---

## 我的建议

如果是你自己用或小团队用，**思路 2（Playwright 代理模式）** 最省心。一个浏览器实例就够了，不用操心 Cookie 刷新、指纹一致性这些问题，而且能实现真多轮对话。

如果要做成服务给很多人用，**思路 4（混合架构）** 最平衡。

你想往哪个方向走？如果确定了方向，我们可以先在浏览器里验证一下关键假设（比如思路 1 的 `__cf_bm` 跨环境可用性），然后再写代码。
